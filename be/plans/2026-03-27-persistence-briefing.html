What Doesn't Work Yet

Gap #1 — Critical

Nothing saves.

Lisa adds a personal note to an alumni profile: "Spoke with Eric at the reception. He's interested in preceptor role. Follow up in April." She closes the tab. Opens it tomorrow. The note is gone.

Same for: engagement checkbox toggles, dismissed triggers, checklist progress, outreach history, alumni invite statuses on projects. All session-only. All lost on refresh.

This isn't a minor gap. This is the difference between a tool and a slideshow.

Gap #2 — High Friction

Email is copy-paste.

Every outreach interaction — the beautifully crafted, voice-matched, trigger-specific draft — ends with Lisa clicking "Copy to Clipboard," opening Outlook, creating a new email, pasting in the To/Subject/Body, reformatting, and clicking Send. That's five extra steps. Every time. For every alumnus.

For a Dean managing 6,000+ alumni relationships, this friction kills adoption.

Gap #3 — Missed Emotional Beat

Lisa's own words aren't in the system.

The notes sections on every profile are empty. The meeting transcript has Lisa saying specific things about specific alumni — her institutional knowledge, her personal impressions. If those words were already in the profiles when she sees the demo, attributed to "Lisa Warren," the reaction shifts from "this is a nice tool" to "this was built for me."

This is a 15-minute task. Highest emotional ROI of anything on the list.

Option A: Cloudflare Worker + D1

How It Works

One Worker. One database. Everything persists.

A Cloudflare Worker is a serverless function that runs at the edge — close to the user, fast, no server to manage. D1 is Cloudflare's SQLite-at-the-edge database. Together they give us a lightweight API that the static site talks to.

Lisa's Browser
↓ fetch('/api/notes', { method: 'POST', body: ... })
Cloudflare Worker (edge function, ~50 lines of code)
↓ INSERT INTO notes (alumni_id, author, text, date) VALUES (...)
D1 Database (SQLite at the edge, 5GB free)
↓ Saved. Persists forever. Available on next visit.
What It Handles

Every action Lisa takes, persisted.

Notes: POST /api/notes — saves personal notes to alumni profiles. Survives refresh, survives tab close, survives device switch.

Engagement toggles: PATCH /api/alumni/:id/engagement — saves checkbox changes.

Outreach history: POST /api/outreach — logs sent emails with date, subject, status.

Trigger dismissals: PATCH /api/triggers/:id — saves dismissed/archived state.

Checklist progress: PATCH /api/projects/:id/checklist — saves task completion.

Alumni invite status: PATCH /api/projects/:id/alumni-status — saves invited/confirmed/declined.

Future-Ready

Same Worker handles email later.

When we're ready for real email integration (Phase 3), we add a POST /api/send-email route to the same Worker. It calls SendGrid, Postmark, or Gmail API. No new infrastructure. The Worker grows with the product.

Authentication? Add a simple token check or Cloudflare Access (zero-trust, free for up to 50 users). Lisa logs in once, stays logged in.

What This Changes in the Plan

Plan Impact

Persistence moves from "Phase 3 someday" to "Phase 2 now."

The original plan deferred persistence. That was wrong. Without persistence, the email integration, the wow-moment notes, the outreach tracking — none of it matters because Lisa can't come back tomorrow and see what she did yesterday.

Revised Phase 2 scope:

1. Cloudflare Worker + D1 backend — persistence for notes, engagement, outreach, triggers, checklists
2. Email from within — mailto: now, Worker-based sending later
3. Lisa's wow moment — pre-populated notes (now they'll persist too)
4. CSV export — "Bring this list to Vegas"
5. Outreach activity stats — Dashboard metrics
6. Visual polish — Rolodex + sidebar confirmation

Cost

The Decision

Do we ship the backend now and give Lisa a tool she can use Monday?
